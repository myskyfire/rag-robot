# 智能文档问答 Agent - 设计文档

## 一、问题拆解与核心挑战

### 1.1 问题拆解
本题本质是一个**文档理解 + RAG 问答 + 可靠性验证**的端到端系统，核心挑战在于：

| 挑战 | 说明 | 应对策略 |
|------|------|---------|
| **扫描 PDF 解析** | 无文本层，需 OCR 识别 | PaddleOCR + 文本后处理 |
| **表格识别** | OCR 表格易错乱 | 表格容错处理 + 独立切块 |
| **OCR 错误** | 形近字、断行、符号错误 | 纠错字典 + 断行合并 + 置信度分级 |
| **无答案问题** | 用户提问超出文档范围 | 检索为空时礼貌拒答 + 幻觉校验 |
| **边界场景** | 模糊提问、跨页内容 | 多路检索 + 风险提示 |

### 1.2 技术选型与取舍

| 组件 | 选型 | 取舍理由 |
|------|------|---------|
| **OCR 引擎** | PaddleOCR | 中文识别准确率高，支持表格识别，本地部署无需 API |
| **向量库** | FAISS | 本地部署、检索高效、无额外服务依赖，适合生产环境 |
| **嵌入模型** | Ollama bge-m3 / DashScope text-embedding-v3 | 支持本地 Ollama 或在线 API，灵活配置 |
| **LLM** | Qwen-Plus | 中文能力强，支持长上下文，API 兼容 OpenAI 格式 |
| **PDF 处理** | PyMuPDF | 纯 Python，支持文本提取、页面转图、表格检测 |

**未选方案说明**：
- 未使用 LangChain/LlamaIndex：本题更关注底层流程理解，而非框架套壳
- 未使用 OpenAI API：国内业务场景，优先选择国产模型

## 二、系统架构设计

### 2.1 整体架构
```
用户问题 → 检索召回 → 答案生成 → 自检风控 → 最终答案
    ↑          ↑          ↑          ↑
  文档输入 → PDF解析 → 分块建库 → 向量检索
```

### 2.2 模块设计（7 个核心模块）

```
模块0: 多格式转PDF     → 统一输入格式
模块1: PDF类型判别     → 选择解析策略（原生文本 vs 扫描 OCR）
模块2: 结构化解析      → 提取正文、条款、表格
模块2.5: OCR后处理     → 断行合并、纠错、置信度分级、残缺检测
模块3: 分块与建库      → 滑动窗口分块，向量入库
模块4: 检索召回        → 向量相似度 + 关键词匹配
模块5: 答案生成        → LLM 生成 + 来源溯源 + 风险提示
模块6: 自检风控        → 无依据校验、幻觉校验、边界处理
```

### 2.3 数据流设计

```
PDF 文件
  ↓
[模块0] 转标准 PDF
  ↓
[模块1] 逐页判别
  ├─ 文本页 → 直接提取文本
  └─ 扫描页 → 转图片 → OCR 识别 → OCR 后处理
  ↓
[模块2] 结构化解析
  ├─ 普通正文 → TextBlock(content_type="text")
  ├─ 编号条款 → TextBlock(content_type="clause", clause_level=N)
  └─ 表格内容 → TextBlock(content_type="table")
  ↓
[模块3] 分块建库
  ├─ 文本块 → 滑动窗口切分 → Chunk(metadata={confidence, is_complete, risk_level})
  └─ 表格块 → 独立存储 → Chunk(content_type="table")
  ↓
[模块4] 检索召回
  ├─ 向量检索 → top_k 相似片段
  └─ 关键词匹配 → 条款编号、专业名词
  ↓
[模块5] 答案生成
  ├─ 构建 prompt（检索内容 + 问题）
  ├─ LLM 生成答案
  └─ 拼接来源信息 + 风险提示
  ↓
[模块6] 自检风控
  ├─ 无依据校验 → 检索为空则拒答
  ├─ 幻觉校验 → 答案与原文比对
  └─ 边界处理 → 模糊提问、跨页内容
```

## 三、关键设计决策

### 3.1 为什么先转 PDF 再解析？
- **统一输入**：支持 doc/xls/ppt/图片/txt 全格式，避免每种格式写解析器
- **标准化**：PDF 是文档交换标准格式，解析工具链成熟

### 3.2 为什么不用 LangChain？
- **透明度**：本题考察对 RAG 流程的理解，而非框架使用能力
- **可控性**：手写流程更容易定位问题、调整策略
- **面试要求**：明确要求"设计合理的 Agent/RAG 流程，而不是简单套壳"

### 3.3 为什么 OCR 后处理独立成模块？
- **职责分离**：OCR 识别与后处理解耦，便于单独测试和优化
- **可复用**：后处理逻辑可独立调用，不依赖解析流程
- **可扩展**：新增纠错规则、置信度策略只需修改后处理模块

### 3.4 分块策略为什么用滑动窗口？
- **语义完整**：固定长度切分可能截断语义，滑动窗口保留上下文
- **简单有效**：原型阶段无需复杂语义分块，滑动窗口足够

### 3.5 为什么要业务适配配置（config/business.py）？
- **配置驱动**：不同业务场景（合同/金融/标准/产品手册）对关键词、分块策略需求不同
- **一键切换**：只需修改 `.env` 中 `BUSINESS_TYPE` 即可切换业务模式
- **零代码扩展**：新增业务场景只需在 `BUSINESS_CONFIGS` 字典中添加新配置项
- **联动检索**：业务关键词自动注入检索器，提升特定领域召回准确率

## 四、完成度说明

- [x] 多格式转 PDF（LibreOffice + PyMuPDF）
- [x] PDF 类型判别（文本量阈值判断）
- [x] 原生文本 PDF 解析（PyMuPDF 提取）
- [x] 扫描 PDF OCR 识别（PaddleOCR）
- [x] OCR 后处理（断行合并、纠错、置信度分级、残缺检测、表格容错）
- [x] 结构化解析（正文、条款、表格）
- [x] 滑动窗口分块
- [x] 向量库构建（FAISS）
- [x] 向量检索 + 关键词匹配
- [x] LLM 答案生成 + 来源溯源
- [x] 自检风控（无依据校验、幻觉校验、OCR 风险提示）
- [x] 日志记录、异常处理、配置管理

### 4.2 未完成功能及原因
- [ ] **图像预处理（OpenCV）**：已移除，PaddleOCR 内置预处理足够，减少依赖
- [ ] **自动化测试脚本**：有手动测试，需补充完整测试用例
- [ ] **演示视频**：需录制

### 4.3 技术债务
- 表格识别依赖 OCR 准确率，复杂表格可能错乱
- 条款层级识别基于正则，可能漏识别特殊格式
- 幻觉校验依赖 LLM 自身判断，可能存在误判

## 五、测试策略

### 5.1 测试用例设计
| 测试类型 | 测试问题 | 预期结果 |
|---------|---------|---------|
| 正文问答 | 本标准规定了什么？ | 正确回答 + 来源页码 |
| 条款问答 | 键的抗拉强度要求？ | 正确回答 + 条款溯源 |
| 表格问答 | 键宽 b 对应的 AQL 值？ | 正确回答 + 表格溯源 |
| 无答案问题 | 键的表面粗糙度要求？ | 礼貌拒答 |
| 模糊问题 | 这个标准怎么样？ | 引导具体提问 |
| 数值比较 | 键宽 10mm 是否合格？ | 基于文档数值判断 |

- 每次修改后运行测试用例，确保不引入新 bug
- 记录测试结果，对比历史数据

## 六、扩展方案

### 6.1 金融场景适配
- 新增金融术语纠错字典
- 优化数值表格解析（金额、利率、百分比）

### 6.2 合同场景适配
- 强化日期、金额、权责条款提取
- 增加合同主体识别

### 6.3 合规文档适配
- 新增法规引用识别
- 增加合规要求提取

### 6.4 扩展方式
核心框架无需修改，仅需：
1. 替换业务配置（关键词、纠错字典、阈值）
2. 调整分块策略（chunk_size、overlap）
3. 优化 prompt 模板（业务特定指令）
