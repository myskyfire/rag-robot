# 多模态模型OCR兜底说明

## 一、功能概述

当PaddleOCR识别置信度过低时，系统可自动调用多模态大模型（如Qwen-VL、GPT-4V等）进行二次识别，作为OCR识别的兜底方案。

## 二、配置方式

在 `.env` 文件中添加以下配置：

```env
# 多模态模型配置（可选，不配置则不启用兜底功能）
MULTIMODAL_API_KEY=your-multimodal-api-key
MULTIMODAL_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
MULTIMODAL_MODEL=qwen-vl-plus

# OCR兜底触发阈值（默认0.6，低于此值触发多模态识别）
OCR_FALLBACK_CONFIDENCE_THRESHOLD=0.6
```

## 三、触发条件

系统在以下情况下自动触发多模态兜底：
1. PaddleOCR识别完成
2. 计算页面平均置信度
3. 平均置信度 < `OCR_FALLBACK_CONFIDENCE_THRESHOLD`
4. 多模态模型已配置（API Key和Base URL不为空）

## 四、工作流程

```
PDF扫描页
  ↓
PaddleOCR识别
  ↓
计算平均置信度
  ↓
置信度 < 阈值？
  ├─ 否 → 使用OCR结果
  └─ 是 → 调用多模态模型
           ↓
       多模态识别成功？
         ├─ 是 → 使用多模态结果
         └─ 否 → 降级使用OCR结果（不崩溃）
```

## 五、⚠️ 风险提示

### 5.1 成本风险
- 多模态模型按token计费，识别一页扫描PDF可能消耗数千token
- 大量文档处理时，费用可能显著增加
- 建议仅在关键文档或OCR置信度极低时使用

### 5.2 性能风险
- 多模态模型识别速度较慢（通常3-10秒/页）
- 依赖外部API，网络不稳定时可能超时
- 大量并发请求可能触发API限流

### 5.3 准确性风险
- 多模态模型对表格、公式等专业内容的识别准确率不一定优于PaddleOCR
- 多模态模型可能产生幻觉，识别出原文不存在的内容
- 输出格式不固定，需要额外解析和校验

### 5.4 依赖风险
- 依赖外部API服务，服务不可用时兜底功能失效
- API密钥泄露可能导致未授权访问和费用损失
- 不同厂商API格式可能不同，需要适配

## 六、降级策略

系统设计了多层降级策略，确保兜底功能异常时不影响主流程：

| 异常场景 | 降级策略 |
|---------|---------|
| 多模态模型未配置 | 跳过兜底，使用OCR结果 |
| API调用失败 | 捕获异常，降级使用OCR结果 |
| 响应解析失败 | 捕获异常，降级使用OCR结果 |
| 响应超时 | 捕获异常，降级使用OCR结果 |
| 多模态置信度更低 | 仍使用OCR结果（取两者中置信度高的） |

## 七、支持的多模态模型

理论上支持任何兼容OpenAI API格式的多模态模型：

| 模型 | Base URL | 说明 |
|------|---------|------|
| Qwen-VL-Plus | https://dashscope.aliyuncs.com/compatible-mode/v1 | 阿里云，中文识别好 |
| Qwen-VL-Max | https://dashscope.aliyuncs.com/compatible-mode/v1 | 阿里云，更强版本 |
| GPT-4V | https://api.openai.com/v1 | OpenAI，英文识别好 |
| Claude Vision | https://api.anthropic.com | Anthropic，需适配API格式 |

## 八、使用建议

### 8.1 推荐场景
- 关键合同、合规文档需要二次校验
- OCR识别置信度极低的页面（<0.5）
- 文档页数较少，成本可控

### 8.2 不推荐场景
- 大批量文档处理（成本高）
- 实时性要求高的场景（速度慢）
- 网络环境不稳定的场景

### 8.3 最佳实践
1. 设置合理的置信度阈值（建议0.5-0.7）
2. 监控API调用费用和响应时间
3. 定期检查兜底触发频率，评估是否需要优化OCR
4. 对多模态识别结果进行抽样人工校验
